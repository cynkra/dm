## Project Overview

The **dm** package provides a framework for working with multiple related data frames or lazy tables in R, allowing users to define, manipulate, and visualize relationships between tables (similar to a database schema). It supports operations such as joins, filtering, and data integrity checks across multiple tables. The package works with local data frames and lazy tables connected to various RDBMS systems (SQLite, PostgreSQL, MSSQL, MariaDB, DuckDB).

## Essential Development Commands

### Load and Test

```r
# Load package for interactive development
pkgload::load_all()

# Run all tests
testthat::test_local(reporter = "check")

# Run tests for a specific file (e.g., test-foo.R)
testthat::test_local(filter = "foo", reporter = "check")

# Run package checks (final validation step)
devtools::check()

# Update roxygen2 documentation
devtools::document()

# Format code to match project style (line width 100, tribble tables)
air format .
```

### Database Testing

```r
# Start test database containers (Docker required)
make db-start

# Test against specific database backends:
make test-sqlite
make test-postgres
make test-mssql
make test-duckdb
make test-maria

# Test against all backends
make -j1 test

# Test with quiet output
make qtest

# Connectivity check
make connect
```

The `Makefile` supports environment variables like `DM_TEST_SRC` (set to "sqlite", "postgres", "mssql", "duckdb", or "maria") to target specific databases when running tests locally.

## Code Architecture

### Package Organization

- **R/** - Main package source code; files organize functionality into logical groups
  - Core functions follow naming logic (see vignette on function naming)
  - Database-specific helpers in `db-*.R` files
  - Error handling via functions in `error-helpers.R`
  - Deprecated functions in `deprecated.R` (never modify these)
- **tests/testthat/** - Test files mirroring source structure (`R/name.R` → `tests/testthat/test-name.R`)
  - `helper-*.R` files contain shared test utilities loaded before tests run
  - `helper-src.R` provides database-agnostic test sources (handles multiple DB backends)
  - `helper-expectations.R` contains custom test expectations
  - `helper-skip.R` contains conditional skip helpers for DB-specific features
  - `_snaps/` directory stores snapshot test results per database variant
- **man/** - Documentation generated by roxygen2 (do not edit `.Rd` files directly)
- **vignettes/** - Package vignettes and articles

### Key Helper Functions

- `dm_for_filter()` - Creates dm objects adapted to the current test database backend
- `my_db_test_src()` - Returns the active test database connection
- `my_test_src_name` - Global variable identifying current database (used for snapshot variants)
- Custom expectations in `helper-expectations.R` for specialized assertions

## Code Style & Conventions

### General Rules

- Follow the [tidyverse style guide](https://style.tidyverse.org) and [tidyverse design guide](https://design.tidyverse.org)
- Use `snake_case` for all new functions
- Indentation: 2 spaces (enforced by `air format .`)
- Line width: 100 characters (configured in `air.toml`)
- Use explicit package prefixes for clarity (e.g., `withr::local_db_connection()`)
- Use meaningful variable names reflecting their context

### Comments

- Prefer expressive code over comments
- Add comments to utility functions that cannot be made immediately obvious
- Focus on explaining "why" and "how", not "what"
- Use line breaks after each sentence in comments

### Documentation (roxygen2 with Markdown)

All exported functions must have:

```r
#' Brief function description
#'
#' Longer description explaining purpose and usage.
#' Each sentence on its own line.
#'
#' @param arg_name Description of argument
#' @return Description of return value
#' @examples
#' # Example code here
#' @export
function_name <- function(arg_name) {
  # ...
}
```

- Run `devtools::document()` after updating roxygen2 comments
- Link to C documentation using `@cdocs tag_name`
- New exported functions must include an example and a concept tag for pkgdown

### New Functions Requirements

All new exported functions must include:

1. Complete roxygen2 documentation with `@examples`
2. Tests (snapshot tests preferred for output-producing functions)
3. A concept tag for pkgdown reference index
4. An experimental badge if applicable: `` `r lifecycle::badge("experimental")` ``

### Error Handling

Errors must use `abort()` through a function defined in `error-helpers.R`. Follow this pattern:

1. Define error class with `dm_error_full()`
2. Create a separate helper function to generate the error message
3. Call abort with the error class and message

This ensures standardized, user-facing error messages.

## Testing Architecture

### Test Structure

- Place new tests near existing tests for the same function
- Add regression tests for bug fixes directly after the last test for that function
- Implement both structured tests and snapshot tests

### Database-Dependent Testing

When a test uses `dm_for_filter()` or `my_db_test_src()` and includes `expect_snapshot()`:

```r
expect_snapshot(result, variant = my_test_src_name)
```

This ensures snapshots are stored in separate directories per database variant.

### Test Helpers

- Use helpers from `helper-skip.R` to conditionally skip tests for unsupported features
- Use custom expectations from `helper-expectations.R`
- Use `withr::` functions for temporary state management in tests

## PR and Commit Style

- PR titles use conventional commits style with backticks for code: `` `function_call()` ``
- PRs are generally squashed; clean history within a PR is not necessary
- Commit messages should be concise and reference related issues
- User-facing changes require a bullet added to the top of `NEWS.md` (follow tidyverse news style)

## Important Notes

- Never modify functions marked as deprecated
- When formatting code, use `air format .` to ensure consistency
- Always run `devtools::check()` as a final validation before committing
- Database tests can be slow; use `DM_TEST_SRC` environment variable for local debugging
- The `scratch/` directory is for exploratory scripts and temporary files (not tracked in package)
